# OpenAPI (Swagger) Documentation for the Backend API

Date: 2026-07-07
Status: Approved, ready for implementation planning

## Problem

GoalPulse Agent's backend (`apps/api`) has 13 endpoints with no formal API
documentation — a judge or reviewer assessing production readiness has no
structured reference describing what the API exposes, what's required (the
`X-API-Key` header on `POST /api/agent/run-once`), or what's rate-limited.

This is the fourth "production readiness" feature this session (after API key
auth, rate limiting, and externally-configured uptime monitoring), constrained to
free tiers with no credit card. This one is documentation only — it must describe
the API exactly as it already behaves, not change any behavior.

## Goals

- A single OpenAPI 3.0 spec (`openapi.yaml`) accurately describing all 13 current
  endpoints: `GET /health`, `/api/matches`, `/api/signals`, `/api/stats`,
  `/api/pnl`, `/api/agent-runs`, `/api/odds-history`, `/api/recent-results`,
  `/api/replay/backtest`, `/api/onchain/validate-stat`, `/api/live/odds-stream`,
  `/api/live/replay-stream`, and `POST /api/agent/run-once`.
- Full schema detail for every endpoint, including the deeply nested
  `/api/replay/backtest` response (summary, timeline, signals, council votes,
  proof hash).
- The `X-API-Key` requirement on `POST /api/agent/run-once` and both rate limits
  (1200/min general, 10/min on run-once) are documented.
- Served live via an interactive Swagger UI at `GET /api/docs`, publicly
  accessible (no API key) so judges can browse and try requests with zero
  friction — this is the most visible way to demonstrate production readiness,
  directly viewable during a demo rather than requiring a judge to open a file.
- Documentation only: no existing route's behavior changes.

## Confirmed behavior (explicitly checked, not assumed)

- `GET /api/docs` will be public exactly like every other GET route — only
  `POST /api/agent/run-once` has `requireApiKey`.
- If a judge uses Swagger UI's "Try it out" on `POST /api/agent/run-once`
  without a key, the real request hits `requireApiKey` before the handler that
  calls `processAgentCycle()` (route is registered as
  `app.post("/api/agent/run-once", runOnceLimiter, requireApiKey, async (...) => {...})`),
  so it fails safely with the existing `401 {"error": "Invalid or missing API key."}`
  rendered in the UI's response panel — no agent cycle runs. The OpenAPI security
  scheme also makes Swagger UI show an "Authorize" field so a key-holder can
  actually exercise the endpoint.

## Design

### File and content

`openapi.yaml` at the repo root (chosen over `apps/api/openapi.yaml` for judge
discoverability — this file exists specifically to demonstrate production
readiness to reviewers browsing the repo, alongside `README.md`,
`SUBMISSION_NOTES.md`, `TECHNICAL_DOCS.md`). Hand-written directly from the real
response shapes already read from `apps/api/src/server.ts` during design — not
generated by any tool, to avoid introducing inaccuracies or requiring unrelated
refactoring (e.g. adopting Zod schemas the codebase doesn't currently have).

Shared types get `components/schemas` entries, `$ref`'d wherever they repeat:
`Match`, `AgentSignal`, `OddsSnapshot`, `AgentRun`, plus the evidence/scores-context
nested objects, matching the shapes already defined in `apps/api/src/types.ts`.

### Auth and rate limits

`components/securitySchemes` defines an `apiKeyAuth` scheme (`X-API-Key` header),
applied only to `POST /api/agent/run-once`'s operation object. Rate limits are
documented in plain text (OpenAPI has no first-class rate-limit field): the global
1200/min limit noted once in the top-level API `description`, and the run-once
endpoint's stricter 10/min limit called out in its own `description` plus a
documented `429` response specific to that endpoint. A general `429` response is
also documented once at the API level for the global limiter.

### Serving mechanism

New route `GET /api/docs` in `server.ts`, using `swagger-ui-express` (renders the
interactive UI) and `yamljs` (`YAML.load(...)` to parse the static file into the
JS object `swagger-ui-express` expects at startup). Both get `@types/` packages
to match this project's existing strict-TypeScript convention (every other
dependency has typed bindings). `/api/docs` is just another GET route — already
covered by the existing global 1200/min limiter, no special-casing needed.

### Testing / verification

No automated test for the YAML content — this is static documentation, and
writing an automated test that re-derives "is this schema accurate" would be
disproportionate effort for a one-time documentation pass. Instead:

1. Run the file through an OpenAPI validator (`npx @redocly/cli lint openapi.yaml`)
   to catch schema/syntax errors before considering the work done.
2. Manually start the dev server, open `/api/docs`, confirm all 13 endpoints
   render correctly.
3. Use "Try it out" on one simple GET endpoint (`/api/matches`) to confirm live
   interactivity actually works end-to-end against the running API.
4. Use "Try it out" on `POST /api/agent/run-once` without a key and confirm the
   401 renders correctly in the UI (per "Confirmed behavior" above).

## Alternatives considered (rejected)

**Auto-generate from JSDoc annotations** (`swagger-jsdoc`, parsing `@openapi` YAML
blocks written as comments above each route). Rejected: adds verbose YAML-in-
comments to `server.ts` (already 837 lines) plus a build-time parsing dependency,
for a one-time documentation pass that doesn't need it.

**Generate from TypeScript types via a schema tool** (e.g. `zod-to-openapi`).
Rejected: would require introducing a whole new Zod validation layer the codebase
doesn't have today just to document existing behavior — disproportionate effort,
and doesn't handle the many ad-hoc computed shapes (like `replay/backtest`) well
since those aren't modeled as typed return values today.
