> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add GitHub Actions CI, pin `apps/api`'s `"latest"` dependencies to caret-range versions, restrict CORS to an explicit allowlist, and add an MIT LICENSE file — the four Tier 1 P1 items that require actual changes (P1-13/P1-14 need none, already documented).

- Verify backend with `npm run test && npm run build` from `apps/api` after the dependency-pin and CORS tasks.
- This is Tier 1 only — do not start Tier 2 after this, report back and wait for review per the user's explicit instruction.

### Task 2: GitHub Actions CI

**Files:**
- Create: `.github/workflows/ci.yml`

- [ ] **Step 1: Add the workflow file**

Create `.github/workflows/ci.yml`:

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  backend:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: apps/api
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '24'
          cache: 'npm'
          cache-dependency-path: apps/api/package-lock.json
      - run: npm ci
      - run: npm run test
      - run: npm run build

  frontend:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: apps/web
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '24'
          cache: 'npm'
          cache-dependency-path: apps/web/package-lock.json
      - run: npm ci
      - run: npm run lint
      - run: npm run build
```

- [ ] **Step 2: Visually re-verify the file**

Read the created file back and confirm indentation is consistent
2-space YAML, matching the block above exactly, and that both jobs'
`working-directory`/`cache-dependency-path` paths correctly point at
`apps/api`/`apps/web` respectively. No local YAML linter is set up in
this project — real end-to-end validation happens automatically once
pushed (GitHub parses and runs the workflow; check the Actions tab
after push).

- [ ] **Step 3: Commit**

```bash
git add .github/workflows/ci.yml
git commit -m "Add GitHub Actions CI workflow"
```

### Task 4: Restrict CORS

**Files:**
- Modify: `apps/api/src/server.ts`

- [ ] **Step 1: Add the allowlist and restrict `cors()`**

Find:

```typescript
app.use(cors());
```

Replace with:

```typescript
const ALLOWED_CORS_ORIGINS = [
  "https://goalpulse-agent.vercel.app",
  "http://localhost:5173",
  "http://127.0.0.1:5173",
];

app.use(cors({ origin: ALLOWED_CORS_ORIGINS }));
```

- [ ] **Step 2: Verify build**

Run from `apps/api`: `npm run build`
Expected: succeeds with no errors.

- [ ] **Step 3: Manual local verification**

Run `npm run dev:once` in `apps/api` in one terminal, then from another terminal:

```bash
curl -s -H "Origin: https://goalpulse-agent.vercel.app" -I http://localhost:4000/health | grep -i access-control-allow-origin
curl -s -H "Origin: https://evil-example.com" -I http://localhost:4000/health | grep -i access-control-allow-origin
```

Expected: the first command's response includes
`Access-Control-Allow-Origin: https://goalpulse-agent.vercel.app`; the
second returns no `Access-Control-Allow-Origin` header at all (the
`cors` middleware simply omits the header for disallowed origins rather
than erroring — this is expected `cors` package behavor, not a bug).
Stop the local API server after checking (exact PID, not pattern-kill).
